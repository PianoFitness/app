# ADR-0028: Exercise History: Configuration-Mirroring Column Schema

**Status:** Accepted

**Date:** 2026-03-28

## Context

ADR-0025 specified an exercise history table with three design choices that were revisited during implementation:

1. **Integer autoincrement primary key** — After ADR-0026 introduced `ExerciseConfiguration` as a unified, identity-free domain model, it was clearer that history entries are append-only events that deserve UUID identifiers. UUID text PKs also keep the schema consistent with `UserProfileTable` and avoid integer overflow in long-lived installations.

2. **Single JSON `configuration` blob** — Storing the complete `ExerciseConfiguration` as a JSON string was the original extensibility strategy. In practice it makes per-column filtering impossible without JSON extraction functions (not well-supported by SQLite in the Flutter/Drift stack), complicates schema type-checking, and adds parse overhead on every read.

3. **Composite `exerciseType` string** — ADR-0025 proposed deriving a human-readable composite key component (`musicalKey + exerciseType`) per practice mode (e.g. `"majorTriad_first"` for chordsByType). This embedded derivation logic in the persistence layer, duplicated `ExerciseConfiguration` semantics, and made querying individual mode parameters harder.

ADR-0026 established `ExerciseConfiguration` as the authoritative domain model for all practice mode parameters. The natural persistence strategy is to mirror those columns 1:1 in the database, exactly as `UserProfileTable` mirrors `UserProfile`. This is the design that was implemented.

## Decision

We implement the exercise history table as a **column-mirroring schema**: every field on `ExerciseConfiguration` that has persistence value gets its own typed Drift column, with the same name and nullable contract as the domain model field. No JSON blobs are stored.

### Primary Key

```dart
/// Unique identifier for this history entry (UUID v4).
TextColumn get id => text()();
```

`id` is a `String` UUID v4 generated by `PracticePageViewModel` at save time using the `uuid` package. This matches the `UserProfileTable` PK convention and supports future offline/distributed record creation without integer key coordination.

### Table Definition

```dart
@DataClassName("ExerciseHistoryTableData")
class ExerciseHistoryTable extends Table {
  TextColumn get id => text()();

  TextColumn get profileId => text().customConstraint(
    "NOT NULL REFERENCES user_profile_table(id) ON DELETE CASCADE",
  )();

  DateTimeColumn get completedAt => dateTime()();

  // ── Exercise configuration columns ─────────────────────────────────────
  TextColumn get practiceMode => text()();
  TextColumn get handSelection => text()();
  TextColumn get musicalKey => text().nullable()();
  TextColumn get scaleType => text().nullable()();
  TextColumn get chordType => text().nullable()();
  BoolColumn get includeInversions =>
      boolean().withDefault(const Constant(false))();
  BoolColumn get includeSeventhChords =>
      boolean().withDefault(const Constant(false))();
  TextColumn get musicalNote => text().nullable()();
  TextColumn get arpeggioType => text().nullable()();
  TextColumn get arpeggioOctaves => text().nullable()();
  TextColumn get chordProgressionId => text().nullable()();

  @override
  Set<Column> get primaryKey => {id};
}
```

**Data class name**: `@DataClassName("ExerciseHistoryTableData")` avoids a name collision with the domain model `ExerciseHistoryEntry`. Drift would otherwise generate `ExerciseHistoryTable` as both the table class and the data class.

### Foreign Key and CASCADE Delete

Drift v2 does not support `references()` with a `TextColumn` FK that carries `ON DELETE CASCADE` via the DSL alone. The constraint is expressed using `customConstraint`:

```dart
TextColumn get profileId => text().customConstraint(
  "NOT NULL REFERENCES user_profile_table(id) ON DELETE CASCADE",
)();
```

This is the same pattern used by `UserProfileTable`-adjacent tables and is verified in the migration test suite.

### Composite Index

The `(profileId, completedAt DESC)` index that enables efficient profile-history queries is declared in the `AppDatabase.from2To3` migration rather than in the table's `get indexes` override. This is because Drift's table-level index API does not support `DESC` ordering on individual columns in all target SQLite versions, and using `issueCustomQuery` in the migration gives full control:

```dart
from2To3: (m, schema) async {
  await m.createTable(schema.exerciseHistoryTable);
  await m.issueCustomQuery(
    "CREATE INDEX IF NOT EXISTS idx_exercise_history_profile_date "
    "ON exercise_history_table (profile_id, completed_at DESC)",
  );
},
```

### Domain Model Separation

The Drift data class `ExerciseHistoryTableData` is private to the application layer. The domain model `ExerciseHistoryEntry` (in `lib/domain/models/practice/`) holds typed enum fields and is constructed from an `ExerciseConfiguration` via a factory:

```dart
factory ExerciseHistoryEntry.fromConfiguration({
  required String id,
  required String profileId,
  required DateTime completedAt,
  required ExerciseConfiguration config,
})
```

The repository implementation (`ExerciseHistoryRepositoryImpl`) maps between the two using enum `.name` strings for all Dart enum fields:

```dart
practiceMode: entry.practiceMode.name,
handSelection: entry.handSelection.name,
musicalKey: Value(entry.musicalKey?.name),
// …
```

Deserialization uses `EnumType.values.byName(string)`, which throws a descriptive error if an unknown name is encountered, providing a clear signal if the stored value and the current enum are out of sync.

### Repository Interface

```dart
abstract class IExerciseHistoryRepository {
  Future<void> saveEntry(ExerciseHistoryEntry entry);
  Future<List<ExerciseHistoryEntry>> getEntriesForProfile(
    String profileId, {
    int? limit,
  });
}
```

`saveEntry` is the write path. `getEntriesForProfile` returns entries most-recent-first, with an optional `limit` cap for lightweight "recent activity" queries.

### Integration Point: Fire-and-Forget

`PracticePageViewModel.initializePracticeSession()` wraps the exercise-completed callback to record history before notifying the UI:

```dart
_practiceSession = PracticeSession(
  onExerciseCompleted: () {
    unawaited(_recordExerciseHistory());
    onExerciseCompleted();
  },
  // …
);
```

`_recordExerciseHistory` is guarded: it silently skips (with a `Logger.warning`) if no active profile is set, and catches/logs all errors without rethrowing, so a history-save failure never blocks or crashes the practice flow.

## Consequences

### Positive

- **Queryable columns**: Each configuration field can be independently filtered, sorted, or aggregated in SQL without JSON extraction functions.
- **Self-documenting schema**: The table definition reads like the domain model; no external mapping table or doc is needed to understand what is stored.
- **No parse overhead on reads**: Enum `.name` → `values.byName()` is faster than `json.decode()` + manual field extraction.
- **Type-safe round-trips**: `values.byName()` validates stored strings against current enum members, surfacing renames early.
- **Privacy by default**: CASCADE delete on the FK guarantees history is removed when a profile is deleted, enforced at the database level without application code.

### Negative

- **Schema migration per new field**: Adding a field to `ExerciseConfiguration` requires a corresponding Drift migration to add the column. This was the trade-off accepted in exchange for queryability.
- **Nullable columns for mode-specific fields**: Most columns are nullable (only `practiceMode` and `handSelection` are always present), so application code must be aware of which columns are meaningful per mode. Drift's type system surfaces this as `String?`; runtime misuse is caught by the domain model.

### Neutral

- **`exerciseType` concept dropped**: The composite string (`musicalKey/exerciseType`) from ADR-0025 does not appear in the schema. Queries that would have filtered on `exerciseType` instead filter on the individual columns (`scaleType`, `chordType`, `arpeggioType`, `chordProgressionId`), which is more precise.
- **No JSON configuration blob**: Reproducing the exact exercise from a history entry uses the stored individual columns to reconstruct an `ExerciseConfiguration`, not JSON deserialization.
